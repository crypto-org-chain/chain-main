# ADR 006: Tiered Lock Rewards (Tier Logic)

## 1. Overview

### 1.1 Goal

Allow users to **lock tokens into a tier** (e.g. 1, 2, or 5 years). Locked tokens are **internal to the tier mechanism**: they can only be delegated or redelegated **within** the tier module. Undelegation is only allowed after the exit commitment has elapsed. **Locking always delegates to a validator** -- there is no locking without delegation. This ensures tier lockers provide security to the network from the moment they lock. Users receive:

- **Base rewards** from normal staking (block rewards, fees) **only when** their tier-locked tokens are delegated to validators.
- **Bonus rewards** as a **fixed APY** on the locked amount (constant bonus APY, e.g. 4% per year), paid from an **external rewards pool**, **only when** the position is delegated to a validator. Bonus stops when the exit commitment has elapsed (no bonus after ExitUnlockAt).

The tier module holds the locked tokens and is the delegator in `x/staking`; users operate via tier-specific messages. **Delegators can also commit an existing delegation to a tier without undelegating** (MsgCommitDelegationToTier: partial or full delegation to one validator; see §5.2).

**TierDelegate and TierRedelegate** move positions between validators. **TierUndelegate is only allowed after the user has triggered exit and the exit commitment has elapsed.** Tier lockers cannot voluntarily undelegate while in the tier or during the exit commitment, so they keep providing security until they leave.

**Add to position:** the owner can **add tokens to an existing position** (same `position_id`) **only when the position is not exiting** (reject if `ExitTriggeredAt != 0`).

**Clear position:** the owner may send **MsgClearPosition** to cancel exit (reset exit timestamps; see §5.6). This allows **MsgAddToTierPosition** and **MsgTierRedelegate** again. It does not withdraw. Rewards are settled before clearing exit so bonus cannot be extended past `ExitUnlockAt` incorrectly (§5.6).

**Withdraw from tier:** the user can **trigger exit at any time**. Once they trigger exit, an **exit commitment** starts (wait X years, depending on tier). **Once that commitment has elapsed**, no more bonus is paid and the user can claim their tokens (after unbonding if delegated). To unbond, the user must have triggered exit and waited until the exit commitment has elapsed before calling **MsgTierUndelegate**.

### 1.2 Non-Goals

- Changing how base delegation rewards are computed or distributed.
- Replacing or duplicating `x/distribution`; we integrate with it.
- Managing inflation or chain-level reward issuance; the external pool is filled by external logic (governance, grants, another module).

## 2. Concepts

| Term | Description |
|------|-------------|
| **Tier** | A lock level defined by an **exit duration** (e.g. 1y, 2y, 5y wait after user triggers exit), a **fixed bonus APY** (per year; e.g. 0.04 = 4%; constant rate), a **minimum lock amount** (smallest amount that can be locked when creating a position in this tier), and a **close-only flag** (see 4.1.1). Bonus is paid at this fixed APY until the exit commitment has elapsed (no bonus after ExitUnlockAt). |
| **Tier-locked tokens** | Tokens sent to the tier module when locking. They are **internal** to the mechanism: they can only be delegated/redelegated via tier messages. **Undelegation is only allowed after the exit commitment has elapsed.** They cannot be used externally (no transfer out except via withdraw from tier). |
| **Lock** | User sends tokens to the tier module and receives a **tier position**. **MsgLockTier** requires a **validator** -- locking always delegates to the specified validator (lock + delegate in one step). Optionally **trigger exit immediately** (exit commitment starts from creation; otherwise the user may trigger exit later via MsgTriggerExitFromTier). The position earns base and **fixed APY** bonus from lock time since it is always delegated. **Tier lockers cannot undelegate** until they have triggered exit **and** the exit commitment has elapsed. The owner can **add** to an existing position **only when not exiting** (reject if exiting). The owner can **cancel exit** with **MsgClearPosition** (reset exit timestamps; see §5.6) to top up or redelegate again. |
| **Base rewards** | Staking rewards from `x/distribution` **only when** tier-locked tokens are delegated to validators (tier module is the delegator). No base rewards accrue while the position is not delegated. |
| **Bonus rewards** | **Fixed APY** on the locked (or delegated) amount, **only while the position is delegated** to a bonded validator. The bonus APY is constant at BonusApy. Accrued bonus = TokensFromShares(DelegatedShares) * BonusApy * (duration in years). Bonus stops at ExitUnlockAt (no bonus after the exit commitment has elapsed). Paid from the **tier rewards pool**. No bonus accrues when the position is not delegated or when the validator is not bonded. |
| **Tier rewards pool** | Module account (or external keeper) holding coins used only to pay APY bonus. Filled by governance, grants, or another module. |
| **Tier position** | A record stored in module state: unique position ID, owner, tier_id, **amount**, optional exit triggered time and exit unlock time, validator and **delegated shares**, base rewards per share snapshot, and last bonus accrual time. A position is always delegated at creation. A position **cannot be broken down**: the full amount is delegated to a single validator as a whole (all-or-nothing). |
| **Withdraw from tier** | Two steps: (1) **Trigger exit** — user can trigger at any time. This starts the **exit commitment** (wait X years, depending on tier). (2) **Claim** — once the exit commitment has elapsed, no more bonus; user can claim tokens (after unbonding if delegated) and position is closed. |
| **Commit delegation to tier** | A delegator can turn **part or all** of an existing (user, validator) delegation into a tier position **without undelegating** (no unbonding). **MsgCommitDelegationToTier(tier_id, validator, amount)** transfers that **amount** (or the corresponding shares) from the user's delegation to the tier module (same validator). **Partial commit is allowed**; the amount must be ≤ the user's delegation to that validator and ≥ the tier's MinLockAmount. One message creates one tier position (one validator; the position itself cannot be broken down). Requires a chain extension to transfer delegation (see §5.2). |

## 3. High-Level Design

- **New module**: `x/tieredrewards` (or `x/lockrewards`).
- **Tier-locked tokens**: Users **lock tokens** into the tier module (transfer to module account). Those tokens are **internal to the tier module**: they can only be **delegated** or **redelegated** via the tier module’s own messages. Undelegation is only allowed after the exit commitment has elapsed (tier lockers cannot voluntarily undelegate while in the tier). They cannot be used outside this mechanism (no external LST).
- **Delegator in staking**: The **tier module account** is the delegator for all tier-locked delegations. Each position is delegated as a whole (full amount to one validator); the module stores per-position the validator address and the **delegated shares** returned by staking when delegating that amount. Base rewards are attributed to position owners when withdrawn. **No rewards (base or bonus) are paid unless the position is delegated** — lockers must delegate to a validator to earn rewards and thus provide security to the network.
- **Bonus**: **Fixed APY** on the locked amount (or on the delegated amount): constant rate BonusApy (no decay). Accrued **only while the position is delegated to a bonded validator**. Bonus stops at ExitUnlockAt. Paid from the tier rewards pool. No multiplier on base rewards.
- **Exit commitment only**: User can trigger exit at any time. When they trigger exit, an exit commitment (X years, depending on tier) starts. Once it has elapsed, no more bonus and they can claim tokens (after unbonding if delegated).
- **Dependencies**: `x/distribution`, `x/staking`, `x/bank`, `x/auth`. Tier module calls staking to delegate/undelegate/redelegate from its module account and distribution to withdraw rewards. For **MsgCommitDelegationToTier** (commit existing delegation without undelegating), the chain must provide a way to **transfer delegation** from one delegator to another (same validator, same shares, no unbonding) — e.g. a staking extension or app-level message (see §5.2). So that **tier lockers can vote** in governance, the tier module exposes voting power per address and the app wires a custom gov tally that includes it (see **§8**).

## 4. State Model

### 4.1 Params (or stored config)

```go
// Tier defines a single tier stored in module state (separate store entries, not inside Params).
type Tier struct {
    Id            uint32          // e.g. 1, 2, 3
    ExitDuration  time.Duration   // after user triggers exit, they must wait this before claim; no bonus after
    BonusApy      sdk.Dec         // Fixed bonus APY (per year); e.g. 0.04 = 4% per year; constant rate; capped at 1.0
    MinLockAmount math.Int        // minimum amount (in bond denom) required when creating a new position in this tier; enforced on MsgLockTier
    CloseOnly     bool            // when true, no new positions can be created in this tier (see §4.1.1)
}

// Params holds module-level parameters (tiers are stored separately, not in Params).
type Params struct {
    TargetBaseRewardsRate sdk.Dec // target base rewards rate for validators; used by BeginBlocker top-up (see §5.8)
}
```

Tiers are managed via governance messages (`MsgAddTier`, `MsgUpdateTier`, `MsgDeleteTier`) and stored as separate keyed entries (`Tiers[tier_id] -> Tier`), not as a list inside `Params`. Bonus payouts use the bond denom.

- **Minimum lock:** Each tier’s `MinLockAmount` must be positive. It is enforced only on **MsgLockTier** and **MsgCommitDelegationToTier** (new positions); **MsgAddToTierPosition** does not require the added amount to meet any minimum.

#### 4.1.1 CloseOnly tier flag

When a tier’s `CloseOnly` field is set to `true` by governance (via `MsgUpdateTier`), the tier is in close-only mode. In this state:

- **Blocked:** `MsgLockTier`, `MsgCommitDelegationToTier`, `MsgAddToTierPosition`, `MsgTierDelegate`, `MsgTierRedelegate`, and `MsgClearPosition` are all rejected for positions in that tier.
- **Allowed:** `MsgTriggerExitFromTier`, `MsgTierUndelegate`, `MsgWithdrawFromTier`, and `MsgClaimTierRewards` remain available so existing positions can exit and close out.

This allows governance to sunset a tier without force-closing existing positions.
- **Fixed APY:** The bonus rate is **constant** at `BonusApy` (e.g. 0.04 = 4% per year). **Formula:** `accrued_bonus = amount_locked × BonusApy × duration_years`, where `duration_years` is the time from `LastBonusAccrual` to the accrual end time, in years. When the position is exiting, accrual end is capped at `ExitUnlockTime` (no bonus after the exit commitment has elapsed). The position stores `LastBonusAccrual`; when exiting, `ExitUnlockTime` caps the accrual period. **SecondsPerYear** is a chain constant (e.g. `365.25 × 24 × 3600`) used to convert duration to years.

#### Definition of BonusApy (with example)

**BonusApy** is the **fixed bonus APY (annual percentage yield, per year)**. It is a decimal (e.g. 0.04 = 4% per year). The rate is constant (no decay).

**Bonus formula:**

`accrued_bonus = Amount * BonusApy * duration_years`

- `duration_years` = (accrual end time - `LastBonusAccrual`) converted to years (e.g. seconds / SecondsPerYear).
- Accrual end time = `min(block_time, ExitUnlockAt)` when the position is exiting; otherwise `block_time`.
- Bonus accrual stops at `ExitUnlockAt` (no bonus for time after the exit commitment has elapsed).

**Example:** `BonusApy = 0.04`, position delegated for 3 years (before ExitUnlockAt): bonus = `Amount * 0.04 * 3` = 12% of amount. For 1,000 tokens, that is 120 tokens of bonus over 3 years.

### 4.2 Tier positions (state)

Tier locks are stored as **state records** in the module. Each record represents one lock and its delegation state. A tier position **cannot be broken down**: when delegated, the **full** `Amount` is delegated to a single validator (all-or-nothing).

```go
// Position: one per lock. Key = position_id (unique).
type Position struct {
    Id                  uint64    // unique ID (e.g. incrementing counter)
    Owner               string    // sdk.AccAddress
    TierId              uint32
    Amount              math.Int  // tokens locked (bond denom); when delegated, this full amount is delegated
    CreatedAtHeight     uint64
    CreatedAtTime       time.Time
    // Exit commitment: once user triggers exit, they must wait until ExitUnlockAt to claim; no bonus after that.
    ExitTriggeredAt     time.Time // zero = not triggered; when set, position is "exiting"
    ExitUnlockAt        time.Time // ExitTriggeredAt + tier.ExitDuration; when user can claim and bonus stops
    // Delegation state: position is either not delegated (Validator empty) or fully delegated to one validator.
    Validator           string    // validator operator address if delegated; empty if not
    DelegatedShares     sdk.Dec   // validator delegation shares for this position (see 4.2.1)
    BaseRewardsPerShare sdk.DecCoins // cumulative rewards-per-share snapshot at creation or last claim (see §5.8)
    LastBonusAccrual    time.Time // last time bonus was accrued (upper bound of last accrual period)
}

// Store layout:
// - Positions:              position_id -> Position
// - PositionsByOwner:       (owner, position_id) -> {} (secondary index for listing by owner)
// - PositionsByTier:        (tier_id, position_id) -> {} (secondary index for listing by tier)
// - PositionsByValidator:   (validator, position_id) -> {} (secondary index for listing by validator)
// - PositionCountByTier:    tier_id -> uint64 (count of positions in each tier)
// - NextPositionId:         next uint64
// - ValidatorRewardRatio:   validator -> ValidatorRewardRatio (cumulative rewards-per-share; see §5.8)
// - UnbondingIdToPositionId / UnbondingIdsByPosition: unbonding_id <-> position_id mapping for slash handling
// - RedelegationIdToPositionId / RedelegationIdsByPosition: redelegation unbonding_id <-> position_id mapping
```

- **Lock into tier:** User sends `MsgLockTier(tier_id, amount, validator)` with required **validator** and optional **trigger_exit_immediately** (exit commitment starts from creation). The amount must be **>=** the tier’s **MinLockAmount**. The tier must not be in close-only mode. Tokens are transferred to the tier module account; the module immediately delegates the full amount to that validator and creates a `Position`. If **trigger_exit_immediately** is set, the module sets `ExitTriggeredAt` and `ExitUnlockAt` so the position is exiting and **exit commitment starts from lock time**; otherwise the user can trigger exit later via MsgTriggerExitFromTier. Multiple positions per owner allowed.
- **Commit existing delegation to tier:** User sends **MsgCommitDelegationToTier(tier_id, validator, amount)** to turn **part or all** of their existing delegation to that validator into a tier position **without undelegating** (no unbonding). **Partial commit is allowed.** Verification: signer must have delegation to that validator ≥ amount; amount ≥ tier MinLockAmount; amount > 0. The specified amount (and corresponding shares) is transferred to the tier module (same validator). One message creates one tier position (one validator; the position cannot be split across validators). Requires a staking/chain extension for delegation transfer (see §5.2).
- **Add to position:** Owner can send `MsgAddToTierPosition(position_id, amount)` to add tokens to an **existing** position. **Not allowed when the position is exiting** — reject if `ExitTriggeredAt != 0`. See §4.6 for what is updated on add.
- **Delegation:** When the user calls `MsgTierDelegate(position_id, validator)`, the module delegates the **full** `Amount` to that validator (position cannot be split). The staking module returns **shares** for that delegation; the tier module stores them in `DelegatedShares` (see §4.2.1). **Only once delegated** do base and bonus rewards accrue. The position's `BaseRewardsPerShare` is set to the validator's current cumulative ratio, and `LastBonusAccrual` is set to block time so bonus accrual starts from delegation time.

#### 4.2.1 Delegated shares calculation

A tier position is always delegated as a **whole**: the full `Amount` is delegated in a single `staking.Delegate` call. The **delegated shares** are the shares returned by the staking module for that delegation.

- **On delegate:** The tier module calls `stakingKeeper.Delegate(ctx, tierModuleAccount, validator, position.Amount)` (bond denom). The staking module converts the token amount to validator delegation shares using the validator’s current exchange rate (tokens per share) and returns the `newShares` created. The tier module sets `position.Validator = validator`, `position.DelegatedShares = newShares`, `position.BaseRewardsPerShare = currentValidatorRatio`, and `position.LastBonusAccrual = block_time` so that base rewards are attributed from the current ratio and bonus accrues only from delegation time.
- **On undelegate:** Allowed **only when the exit commitment has elapsed** (`block_time >= ExitUnlockAt`). The tier module calls `stakingKeeper.Undelegate(ctx, tierModuleAccount, position.Validator, position.DelegatedShares)` -- the **full** shares for this position. No partial undelegate; the position’s entire delegation is unbonded. While the exit commitment has not elapsed, **MsgTierUndelegate** is rejected so that tier lockers stay delegated and keep providing security. An unbonding-to-position mapping is recorded for slash handling.
- **On redelegate:** The tier module calls `stakingKeeper.BeginRedelegate(ctx, tierModuleAccount, position.Validator, dstValidator, position.DelegatedShares)`. The full shares are moved to the destination validator. The position’s `Validator` is updated to the destination, `DelegatedShares` is set to the **new shares** on the destination validator, and `BaseRewardsPerShare` is set to the destination validator’s current cumulative ratio. A redelegation-to-position mapping is recorded for slash handling.

Because each position is a single delegation of a fixed amount, there is a 1:1 mapping between a tier position and one delegation entry in staking (for that validator). Base rewards for that delegation are attributed entirely to that position’s owner when they call `MsgClaimTierRewards`.

### 4.3 Withdraw from tier (two phases)

The user can **trigger exit at any time**. Withdraw is a **two-phase** process:

**Phase 1 — Trigger exit**

- **Message:** `MsgTriggerExitFromTier(position_id)`. Auth: signer must be `Position.Owner`. Reject if already exiting (`ExitTriggeredAt != 0`). Set `ExitTriggeredAt = block_time`, `ExitUnlockAt = ExitTriggeredAt + tiers[tier_id].ExitDuration`. Position is now in **exiting** state.
- User must wait until `block_time >= ExitUnlockAt` before they can claim. During the exit commitment period, bonus can still accrue. Once exit commitment has elapsed (`block_time >= ExitUnlockAt`), **no more bonus** is paid for this position.

**Phase 2 — Claim tokens**

- Allowed only when `block_time >= position.ExitUnlockAt` (exit commitment elapsed). Reject otherwise.
- **Message:** `MsgWithdrawFromTier(position_id)`. Auth: signer must be `Position.Owner`. If position is delegated, require no active delegation and no unbonding entries (user must have undelegated and waited for unbonding). Transfer `Amount` from tier module to owner, delete position, emit event.

So: **lock -> user may trigger exit anytime -> wait X years (exit commitment, per tier) -> no more bonus -> claim tokens** (see **MsgWithdrawFromTier**, §5.7).

### 4.4 Tier rewards pool

- Module account in tiered rewards module named `rewards_pool`. To be funded from external parties via bank send.

### 4.5 How tier rewards are tracked and calculated

Tier rewards consist of **base rewards** (from staking/distribution) and **bonus rewards** (fixed APY from the tier pool). They are tracked and calculated as follows.

#### Base rewards (staking)

- **Where they are tracked:** In `x/distribution`, not in the tier module. The distribution module keeps:
  - **Validator historical rewards:** cumulative reward ratio per validator per period (updated when a period is closed and when rewards are allocated).
  - **Delegator starting info:** per (delegator, validator) the starting period and the delegator's cumulative ratio cursor, so that delegator reward = shares × (current_ratio − starting_ratio).
- **Tier's use:** The **tier module account** is the delegator. Each tier position contributes to a shared delegation on its validator. When the module needs to attribute base rewards to a specific position, it uses a **cumulative rewards-per-share** mechanism (see §5.8): for each validator, the module maintains `ValidatorRewardRatio` (a running total of rewards-per-share). Each position stores a `BaseRewardsPerShare` snapshot at creation or last claim. When the owner claims rewards, the position's base reward is `DelegatedShares * (currentRatio - BaseRewardsPerShare)`.
- **Per-position base state in tier:** Unlike the original design which relied solely on distribution, the implementation stores per-position `BaseRewardsPerShare` and per-validator `ValidatorRewardRatio` to efficiently attribute base rewards across multiple positions sharing the same delegation.

#### Bonus rewards (fixed APY)

- **Where they are tracked:** In the tier module, **per position**:
  - **`LastBonusAccrual`** (time): the last time up to which bonus has been accrued and paid. The next accrual period runs from `LastBonusAccrual` to the accrual end time.
  - **`ExitUnlockAt`** (when exiting): used to cap the accrual end time so no bonus is paid after the exit commitment has elapsed.
- **Calculation (fixed APY):** When paying tier rewards (e.g. in `MsgClaimTierRewards`), **the position must be delegated** (`Validator` set); otherwise skip bonus. **Single formula:**
  1. **Accrual end:** `accrual_end = block_time`; if the position is exiting and `block_time > ExitUnlockAt`, set `accrual_end = ExitUnlockAt`.
  2. **Duration in years:** `duration_years = (accrual_end - LastBonusAccrual)` in years (e.g. seconds / SecondsPerYear).
  3. **Accrued bonus:** `accrued_bonus = Amount * BonusApy * duration_years`. For normal user-driven claim paths, if the bonus pool cannot cover the full amount, **fail the claim atomically** and do not update `LastBonusAccrual` so the user can retry later after the pool is replenished. On successful payment, send the full amount to the position owner and set `LastBonusAccrual = accrual_end`.
  4. **Bonus uses delegated token value:** The bonus is computed on the token value of the position's `DelegatedShares` (via `validator.TokensFromShares`), not on a raw stored amount. This ensures slashing is automatically reflected in bonus calculations.
- **When accrual starts:** When the position is **first delegated**, set `LastBonusAccrual = block_time`. No bonus accrues while the position is not delegated.
- **When accrual stops:** (1) Position not delegated -> no bonus. (2) Position exiting -> no bonus after `ExitUnlockAt` (enforced by capping `accrual_end` at `ExitUnlockAt`). (3) Validator not bonded -> no bonus (yields zero).
- **Validator lifecycle hooks:** When a validator begins unbonding, the implementation settles the **final bonded-period bonus** for all positions on that validator inside `AfterValidatorBeginUnbonding` with `forceAccrue=true` (because the SDK changes the validator status to Unbonding before firing this hook). When a validator becomes bonded again, `AfterValidatorBonded` resets `LastBonusAccrual` to block time for all positions on that validator so bonus only accrues from when the validator is bonded again. When a validator is removed, `AfterValidatorRemoved` cleans up the validator's reward ratio entry.
- **Slashing hooks:** Before reducing `Amount` for a slash, `BeforeValidatorSlashed` settles pending rewards for all positions on that validator, then reduces each position's `Amount` (and `DelegatedShares`) proportionally to the slash fraction. Additional hooks handle unbonding delegation slashes and redelegation slashes by updating the affected position's `Amount` and `DelegatedShares` respectively.
- **Insufficient bonus pool during hook settlement:** This is intentionally different from normal user-driven claims. For liveness, hook-driven settlement must not fail validator state transitions. The implementation still advances `LastBonusAccrual` to the hook settlement time even if the bonus transfer cannot be paid. The unpaid bonus is logged and forfeited. No per-position unpaid-bonus debt is tracked.
- **Other user-driven settlement paths:** The same "no partial bonus payout" rule applies to user-driven lifecycle messages that settle rewards before mutating the position (for example, undelegate / redelegate / add-to-position / clear-position flows that claim-and-refresh first). If bonus settlement is required on those paths and the pool is insufficient, the message fails rather than partially paying bonus.

#### Summary

| Reward type | Tracked in | Tier module state used | When calculated |
|------------|------------|------------------------|------------------|
| Base       | `x/distribution` (validator historical rewards, delegator starting info); tier module tracks cumulative rewards-per-share per validator (`ValidatorRewardRatio`) and per-position snapshot (`BaseRewardsPerShare`) | `position.Validator`, `position.DelegatedShares`, `position.BaseRewardsPerShare` | On `MsgClaimTierRewards` (and before any delegation-mutating operation): **only if position is delegated**; withdraw from distribution, update `ValidatorRewardRatio`, compute `reward = DelegatedShares * (currentRatio - BaseRewardsPerShare)`, send to owner. No base rewards when not delegated. |
| Bonus      | Tier module: `position.LastBonusAccrual`; when exiting, `ExitUnlockAt` to cap accrual | Same | On `MsgClaimTierRewards` (and before any delegation-mutating operation): **only if position is delegated**. **Fixed APY:** accrued = TokensFromShares(DelegatedShares) * BonusApy * (duration in years); cap accrual_end at ExitUnlockAt. If the bonus pool cannot cover the full amount, the user-driven claim fails atomically. On success, pay the full amount to the owner and set `LastBonusAccrual = accrual_end`. No bonus when not delegated or when validator is not bonded. |

### 4.6 When the tier position must be updated

The tier position is a state record that must stay consistent with staking and with user actions. The following cases require **updating** the position (or deleting it).

| Trigger | What to update | Notes |
|--------|----------------|--------|
| **MsgLockTier** (optional args) | Always: `Validator`, `DelegatedShares`, `BaseRewardsPerShare`, `LastBonusAccrual`. If **trigger_exit_immediately**: `ExitTriggeredAt`, `ExitUnlockAt` | Lock creates the position and always delegates. Optional trigger exit at creation (see §5.1). |
| **MsgCommitDelegationToTier** | Creates new position: `Id`, `Owner`, `TierId`, `Amount` = committed amount, `Validator` = validator, `DelegatedShares` = transferred shares for that amount, `BaseRewardsPerShare` = current validator ratio, `LastBonusAccrual` = block_time, `ExitTriggeredAt` = 0, `ExitUnlockAt` = 0 | Transfers **amount** (partial or full) of user’s delegation to specified validator to tier module (no unbonding). Verification: amount <= user’s delegation to validator, amount >= MinLockAmount, amount > 0 (see §5.2). |
| **MsgTierDelegate** | `Validator`, `DelegatedShares`, `BaseRewardsPerShare`, `LastBonusAccrual` | Set to the chosen validator and the shares returned by staking. Set `BaseRewardsPerShare = currentValidatorRatio` and `LastBonusAccrual = block_time` so rewards accrue only from delegation time. Rejected if the tier is in close-only mode. |
| **MsgAddToTierPosition** | See “Add to position” below | **Not allowed when exiting** (reject if `ExitTriggeredAt != 0`). **Not allowed when tier is close-only.** Updates listed in next subsection. |
| **Unbonding completes** (after MsgTierUndelegate) | `Validator` -> empty, `DelegatedShares` -> 0, `BaseRewardsPerShare` -> cleared, `LastBonusAccrual` -> cleared | When the tier module’s unbonding for this position’s shares completes, the position is no longer delegated. Clear delegation state so the position is “undelegated”; **no rewards (base or bonus) accrue until the owner delegates again**. Unbonding-to-position mappings are cleaned up by the unbonding completion hook. |
| **Redelegation completes** | Redelegation-to-position mappings cleaned up by completion hook | Validator and shares are updated at redelegation initiation time (not completion). Redelegation mappings are removed on completion. |
| **MsgTriggerExitFromTier** | `ExitTriggeredAt`, `ExitUnlockAt` | Position enters “exiting” state; bonus stops at `ExitUnlockAt` (no bonus after exit commitment elapsed). |
| **MsgClearPosition** | `ExitTriggeredAt` -> 0, `ExitUnlockAt` -> 0; if clearing after exit lock duration elapsed, also resets `LastBonusAccrual` to block time | Cancels exit commitment. Rewards are settled unconditionally (whether delegated or not) **before** clearing so bonus is not retro-accrued past `ExitUnlockAt`. **Rejected if the tier is in close-only mode.** Idempotent when already not exiting. |
| **MsgClaimTierRewards** | `BaseRewardsPerShare` -> current validator ratio, `LastBonusAccrual` -> accrual_end | So rewards are not double-counted for the same period. |
| **Slashing** | `Amount` (and `DelegatedShares`) | When the validator or the tier module’s delegation is slashed, the token value of the delegation drops. The position’s `Amount` is reduced proportionally to the slash fraction, and `DelegatedShares` is updated if shares are burnt. The implementation settles pending rewards before slashing via `BeforeValidatorSlashed`. |
| **MsgWithdrawFromTier** (claim) | Position **deleted** | Not an update; the position is removed after tokens are sent to the owner. |
| **Validator leaves active set** (jailed, unbonding, removed) | No mandatory update | The delegation still exists in staking (the validator may be jailed, unbonding, or unbonded). The position’s `Validator` and `DelegatedShares` remain valid; the user can call **MsgTierRedelegate** to move to another validator or **MsgTierUndelegate** to unbond. No new block rewards are earned while the validator is inactive. **No automatic position update** is required. Optional: if the chain implements auto-undelegate when a validator is removed (e.g. for safety), then when that unbonding completes the position is updated as in “Unbonding completes” above. |

**Summary:** Delegation state (`Validator`, `DelegatedShares`) is updated on delegate (including at lock via MsgLockTier), on unbonding completion (only after exit-triggered undelegate), and on redelegation initiation. Exit state is updated on trigger exit (including optional trigger at lock via MsgLockTier). **Undelegate is only allowed after trigger exit** so tier lockers stay delegated until they leave. Reward state (`BaseRewardsPerShare`, `LastBonusAccrual`) is updated on `MsgClaimTierRewards` and before any delegation-mutating operation. **Add to position** updates `Amount` (and delegation state if already delegated). **Slashing** requires updating `Amount` (and `DelegatedShares`) so the position reflects the current value of the delegation and bonus/payout are correct.

#### Add to position: what is updated (MsgAddToTierPosition)

When the owner adds tokens to an existing position (`position_id`), the following are updated. **Add is only allowed when the position is not exiting** (reject if `ExitTriggeredAt != 0`). **Add is rejected when the tier is in close-only mode.**

| Field / action | Update |
|----------------|--------|
| **Amount** | Increase by the added `amount`: `position.Amount += amount`. Tokens are transferred from the owner to the tier module before this update. |
| **Validator** | Unchanged (position remains delegated to the same validator or remains undelegated). |
| **DelegatedShares** | **If position is delegated** (`Validator != ""`): claim rewards first, then call `staking.Delegate(ctx, tier_module_account, position.Validator, amount)` to delegate the **new** tokens to the same validator. Staking returns `newShares`. Set `position.DelegatedShares += newShares`. **If position is not delegated:** no change; just increase Amount. |
| **LastBonusAccrual** | When delegated, rewards are claimed before adding (settle bonus to now, set `LastBonusAccrual = block_time`), so the new tokens accrue bonus only from add-time onward. |
| **CreatedAtHeight**, **CreatedAtTime** | Unchanged (position creation time is preserved). |
| **ExitTriggeredAt**, **ExitUnlockAt** | Unchanged (add is only when not exiting). |
| **Owner**, **TierId**, **Id** | Unchanged. |

## 5. Message Flows

### 5.1 Lock into tier

**Message:** `MsgLockTier(tier_id, amount, validator)` with optional field:
- **trigger_exit_immediately** (optional, bool): if true, set exit state at creation so the **exit commitment starts from lock time**; otherwise the position is not exiting and the user may trigger exit later via MsgTriggerExitFromTier.

```
User -> MsgLockTier(tier_id, amount, validator [, trigger_exit_immediately])
  -> Validate tier_id exists and tier is not close-only; amount > 0; denom = bond denom; amount >= tiers[tier_id].MinLockAmount
  -> Validate validator exists and is in active set (or allow and let staking reject)
  -> Bank.SendCoinsFromAccountToModule(owner, tieredrewards.ModuleName, amount)
  -> Update ValidatorRewardRatio for this validator (withdraw distribution rewards first)
  -> Staking.Delegate(ctx, tier_module_account, validator, amount)
  -> position_id = NextPositionId; NextPositionId++
  -> Set Position: position_id, owner, tier_id, Amount=amount, CreatedAtHeight, CreatedAtTime,
       ExitTriggeredAt=0, ExitUnlockAt=0, Validator=validator, DelegatedShares=newShares,
       BaseRewardsPerShare=currentValidatorRatio, LastBonusAccrual=block_time

  -> If trigger_exit_immediately:
       position.ExitTriggeredAt = block_time
       position.ExitUnlockAt = block_time + tiers[tier_id].ExitDuration

  -> Index by owner (PositionsByOwner), tier (PositionsByTier), validator (PositionsByValidator)
  -> Emit event (position_id, owner, tier_id, amount, validator [, exiting])
```

Tokens are held by the tier module and delegated immediately to the specified validator. If `trigger_exit_immediately` was set, the position is already exiting and the exit commitment runs from creation.

### 5.2 Commit existing delegation to tier

**Message:** `MsgCommitDelegationToTier(tier_id, validator, amount)`. Allows a delegator to turn **part or all** of an existing delegation (their tokens already delegated to a validator) into a tier position **without undelegating** — no unbonding, no transfer of tokens through the bank. **Partial commit is allowed**: the user specifies the `amount` to commit (bounded by their delegation to that validator).

**Verification (all required):**
1. **tier_id** exists and **validator** exists (and is in active set if the chain requires it).
2. **amount** > 0; **amount** in bond denom.
3. **amount** ≥ tier's **MinLockAmount** for the given tier_id.
4. Signer has a delegation to **validator**; let `user_delegation_tokens` = token value of that delegation (e.g. from staking `GetDelegation(signer, validator)` and `TokensFromShares`).
5. **amount** ≤ **user_delegation_tokens** (cannot commit more than the user has delegated to this validator).

If any check fails, reject the message.

**Chain requirement:** The staking module (or app) must expose a way to **transfer delegation** from one delegator to another for the same validator without unbonding — e.g. `TransferDelegation(ctx, from_delegator, to_delegator, validator, amount)` or `(..., shares)` which decreases (from_delegator, validator) and increases (to_delegator, validator) by that amount/shares. If the chain does not support this, MsgCommitDelegationToTier is not available (or must be implemented via another mechanism).

```
User -> MsgCommitDelegationToTier(tier_id, validator, amount)
  -> Auth: signer == owner (the delegator)
  -> Validate tier_id exists and tier is not close-only; validator exists
  -> Validate amount > 0; denom = bond denom; amount >= tiers[tier_id].MinLockAmount
  -> Query staking: get owner's delegation to validator (shares, token amount)
  -> Require delegation exists
  -> Require amount <= owner's delegation token amount to this validator (partial commit allowed)
  -> Update ValidatorRewardRatio for this validator
  -> Staking.TransferDelegation(ctx, owner, tier_module_account, validator, amount_or_shares)  // chain extension
  -> position_id = NextPositionId; NextPositionId++
  -> Set Position: position_id, owner, tier_id, Amount = amount,
       Validator = validator, DelegatedShares = transferred shares,
       BaseRewardsPerShare = currentValidatorRatio, LastBonusAccrual = block_time,
       ExitTriggeredAt = 0, ExitUnlockAt = 0, CreatedAtHeight, CreatedAtTime
  -> Index by owner (PositionsByOwner), tier (PositionsByTier), validator (PositionsByValidator)
  -> Emit event (position_id, owner, tier_id, validator, amount)
```

No bank transfer: the delegation ownership for that amount/shares moves from (owner, validator) to (tier_module, validator). The resulting tier position is already delegated to that validator; base and bonus rewards accrue from commit time. The position cannot be broken down (same as all tier positions): it is one position, one validator, one amount per message; the user can send multiple messages to commit more later or to commit to other validators.

### 5.3 Add to existing position

```
User -> MsgAddToTierPosition(position_id, amount)
  -> Auth: signer == Position.Owner
  -> Load Position; require position exists; reject if position is exiting (ExitTriggeredAt != 0); reject if tier is close-only
  -> Validate amount > 0; denom = bond denom
  -> Bank.SendCoinsFromAccountToModule(owner, tieredrewards.ModuleName, amount)
  -> If position is delegated (Validator != “”):
       Claim rewards for position (base + bonus) before adding
       newShares = Staking.Delegate(ctx, tier_module_account, position.Validator, amount)
       position.DelegatedShares += newShares
  -> Else:
       position.Amount += amount
  -> Save Position
  -> Emit event (position_id, owner, amount_added, new_total)
```

Only the owner can add. Add is **not allowed when the position is exiting** (reject if `ExitTriggeredAt != 0`) and **not allowed when the tier is in close-only mode**. See §4.6 “Add to position: what is updated” for the full list of fields updated.

### 5.4 Tier delegate / undelegate / redelegate (internal only)

Tier-locked tokens can only be staked **within** the tier mechanism. The **tier module account** is the delegator in `x/staking`. A tier position **cannot be broken down**: delegation is always the **full** amount of the position. **Tier lockers cannot undelegate until the exit commitment has elapsed.** This keeps them delegated and providing security to the network until the commitment period is over.

**MsgTierDelegate(position_id, validator)** -- Note: in the current implementation, delegation always happens at lock time (`MsgLockTier` and `MsgCommitDelegationToTier` both require a validator). `MsgTierDelegate` exists in the proto definition for delegating a position that became undelegated after an undelegate cycle, but new positions are always created in a delegated state. It is rejected when the tier is in close-only mode, and rejected when the exit lock duration has already elapsed.

```
  -> Auth: signer == Position.Owner
  -> Load Position; require position not already delegated (Validator == "")
  -> Require tier is not close-only
  -> Require exit lock duration has not elapsed (if exit was triggered)
  -> Require position Amount > 0
  -> Staking.Delegate(ctx, tier_module_account, validator, position.Amount)
  -> Staking returns newShares (shares issued for this delegation)
  -> Update position: Validator = validator, DelegatedShares = newShares; BaseRewardsPerShare = currentValidatorRatio; LastBonusAccrual = block_time (rewards start when delegated)
  -> Emit event (position_id, owner, validator)
```

**MsgTierUndelegate(position_id)**

```
  -> Auth: signer == Position.Owner
  -> Load Position; require position is delegated (Validator != ""); require position has triggered exit (ExitTriggeredAt != 0); require exit commitment has elapsed (block_time >= ExitUnlockAt)
  -> Claim rewards for position (base + bonus)
  -> Staking.Undelegate(ctx, tier_module_account, position.Validator, position.DelegatedShares)
  -> Record unbonding-to-position mapping for slash handling
  -> Update position Amount to the return amount from undelegation; clear delegation state
  -> Emit event (position_id, owner, completion_time, unbonding_id)
```

**MsgTierRedelegate(position_id, dst_validator)**

Redelegation is allowed during exit-in-progress (before the exit lock duration has elapsed). It is **blocked only after** the exit lock duration has elapsed (`block_time >= ExitUnlockAt`), because at that point the position should be undelegated, not moved. It is also rejected when the tier is in close-only mode, or when the position Amount is zero, or when the destination validator is the same as the source.

```
  -> Auth: signer == Position.Owner
  -> Load Position; require position is delegated (Validator != "")
  -> Require exit lock duration has NOT elapsed (if exit was triggered)
  -> Require tier is not close-only
  -> Require position Amount > 0
  -> Require dst_validator != position.Validator
  -> Claim rewards for position (base + bonus)
  -> Snapshot destination validator's reward ratio
  -> Staking.BeginRedelegate(ctx, tier_module_account, position.Validator, dst_validator, position.DelegatedShares)
  -> Record redelegation-to-position mapping for slash handling
  -> Update position: Validator = dst_validator, DelegatedShares = newShares, BaseRewardsPerShare = dstValidatorRatio
  -> Emit event (position_id, owner, src_validator, dst_validator, completion_time, unbonding_id)
```

These tokens **cannot** be used outside the tier module (no external LST); they are internal to this mechanism.

### 5.5 Trigger exit from tier

```
User -> MsgTriggerExitFromTier(position_id)
  -> Auth: signer == Position.Owner
  -> Load Position; reject if already exiting (ExitTriggeredAt != 0)
  -> ExitTriggeredAt = block_time; ExitUnlockAt = ExitTriggeredAt + tiers[tier_id].ExitDuration
  -> Update Position
  -> Emit event (position_id, owner, ExitUnlockAt)
```

User stays in tier (earning base + bonus) until they trigger exit. After triggering, they must wait until `ExitUnlockAt` to claim. Once that time has passed, no more bonus.

### 5.6 Clear position (cancel exit commitment)

**Message:** `MsgClearPosition(owner, position_id)`. Signer must be `owner` and must match `Position.Owner`. **Response:** returns `position_id` (echo).

The owner may **cancel** the exit commitment without withdrawing: reset `ExitTriggeredAt` and `ExitUnlockAt` to zero so the position is **not** exiting. **Rejected if the tier is in close-only mode.** **Effects:**

- **MsgAddToTierPosition** is allowed again (add was rejected while `ExitTriggeredAt != 0`).
- **MsgTierRedelegate** is allowed again while delegated (redelegate is blocked only after exit lock duration has elapsed).
- **MsgTierUndelegate** requires exit commitment to have elapsed; after clear, the owner must call **MsgTriggerExitFromTier** again and wait for the commitment period before undelegating.

**Auth:** Signer must be `Position.Owner`.

**Settle rewards unconditionally before clearing:** The implementation **always** runs reward settlement (claim base + bonus) before clearing exit, regardless of whether the position is delegated. This ensures bonus cannot be retro-accrued for wall-clock time past `ExitUnlockAt` after the cap is removed by clearing exit. When clearing a position that has completed its exit lock duration, `LastBonusAccrual` is reset to block time to prevent extra bonus accrual.

**Additional validation when exit lock duration has elapsed:** If the position has completed its exit lock duration (`block_time >= ExitUnlockAt`), the implementation also requires that the position is not unbonding and is delegated. This prevents clearing exit on a position that is already in the withdraw path.

**Undelegation mappings:** Clearing exit does **not** remove staking unbonding / redelegation ID mappings; slashing callbacks that key off unbonding IDs remain valid.

```
User -> MsgClearPosition(owner, position_id)
  -> Auth: signer == owner == Position.Owner
  -> Load Position; require position exists
  -> Require tier is not close-only
  -> If exit lock duration has elapsed: require position is not unbonding and is delegated
  -> Settle base + bonus for this position (cap bonus accrual at ExitUnlockAt when applicable)
  -> If position had not triggered exit: return early (no-op)
  -> Clear exit: ExitTriggeredAt = 0, ExitUnlockAt = 0; reset LastBonusAccrual = block_time
  -> Save Position
  -> Emit event (position_id, owner, tier_id)
  -> Response: echo position_id
```

### 5.7 Withdraw from tier (claim after exit commitment)

```
User -> MsgWithdrawFromTier(position_id)
  -> Auth: signer == Position.Owner
  -> Load Position; require block_time >= position.ExitUnlockAt (exit commitment elapsed). If ExitTriggeredAt is zero, reject (must trigger exit first).
  -> If delegated: reject (must have undelegated first)
  -> Require no pending unbonding entries for this position
  -> Bank.SendCoinsFromModuleToAccount(tier_module, owner, position.Amount)
  -> Delete Position(position_id)
  -> Emit event (position_id, owner, amount)
```

No bonus is paid after `ExitUnlockAt`; this message only transfers tokens and removes the position.

### 5.8 Claim tier rewards (base + fixed APY bonus)

See **§4.5** for how base and bonus rewards are tracked and calculated.

**Flow:** The module **withdraws the base reward** (from `x/distribution` into the tier module), updates the cumulative rewards-per-share ratio for the validator, then **computes and sends** each position’s share to its owner.

```
User -> MsgClaimTierRewards(position_id)
  -> Auth: signer == Position.Owner
  -> Load Position; if not delegated, return zero rewards

  --- Phase 1: Update validator reward ratio and compute base reward ---
  -> Withdraw distribution rewards for this validator (once per block, deduplicated via transient store)
  -> Update ValidatorRewardRatio[validator]: currentRatio += (withdrawn_rewards / total_module_shares_on_validator)
  -> Compute position base reward: reward = position.DelegatedShares * (currentRatio - position.BaseRewardsPerShare)
  -> Send truncated reward coins from tier module to position owner
  -> Set position.BaseRewardsPerShare = currentRatio

  --- Phase 2: Pay bonus (fixed APY) ---
  -> Bonus (fixed APY): pay only for the accrual interval [LastBonusAccrual, accrual_end]. Compute as follows:
       1. accrual_end_time = block_time
          If position.ExitTriggeredAt != 0 and block_time > position.ExitUnlockAt, set accrual_end_time = position.ExitUnlockAt (no bonus after exit commitment).
       2. accrual_interval_seconds = (accrual_end_time - position.LastBonusAccrual) in seconds (must be >= 0; if 0, skip bonus).
       3. duration_years = accrual_interval_seconds / SecondsPerYear  (chain constant: 31_557_600 for 365.25 days).
       4. tokens = validator.TokensFromShares(position.DelegatedShares)
       5. rewards = tokens * tier.BonusApy * duration_years
       6. If `rewards_pool_balance < rewards`, fail the user-driven claim path and do not update `LastBonusAccrual`.
       7. Otherwise, send the full `rewards` amount from the tier rewards pool to `position.Owner`.
  -> Set position.LastBonusAccrual = accrual_end_time
  -> Emit event (position_id, owner, base_amount, bonus_amount)
```

- **Order of operations:** Base rewards are withdrawn from distribution and the ratio is updated **before** any per-position computation. The module then computes each position’s share from the ratio delta and sends it to the owner, followed by bonus from the tier pool.
- **Fixed APY:** Bonus uses the token value of delegated shares (via `validator.TokensFromShares`), not a raw stored amount. Accrual is capped at `ExitUnlockAt` when exiting. If the pool balance is insufficient, the user-driven claim fails atomically so the user can retry later; there is no partial payout on this path.

#### Base reward attribution: cumulative rewards-per-share (ValidatorRewardRatio)

In standard Cosmos SDK staking, there is **one delegation per (delegator, validator)**. The tier module is a single delegator, so it has **one delegation per validator** (all positions on that validator are aggregated into one delegation in staking). Base rewards are therefore accrued once per validator. The implementation uses a **cumulative rewards-per-share** approach to attribute rewards to individual positions efficiently.

**Mechanism:**

1. **`ValidatorRewardRatio[V]`** stores the cumulative `rewards-per-share` for validator `V`. This is a running total: each time distribution rewards are withdrawn for `V`, the delta (`withdrawn_rewards / total_module_shares_on_V`) is added to the ratio.

2. Each position stores **`BaseRewardsPerShare`**: a snapshot of the validator’s cumulative ratio at the time the position was created or last claimed. When the owner claims rewards, the position’s share is:

   ```
   reward = position.DelegatedShares * (ValidatorRewardRatio[V].CumulativeRewardsPerShare - position.BaseRewardsPerShare)
   ```

3. After claiming, the position’s `BaseRewardsPerShare` is set to the current ratio, so the same rewards are not paid again.

**Deduplication:** Distribution withdrawals are deduplicated per validator per block using a transient store key (`LastRewardsWithdrawalBlock[V]`). If the validator’s rewards were already withdrawn in this block, the current ratio is returned without calling distribution again.

**When `updateBaseRewardsPerShare` is called:** Before **any** operation that changes the module’s delegation shares for a validator (delegate, undelegate, redelegate, slash). This ensures the ratio correctly reflects the rewards earned under the pre-mutation share count.

## 7. Integration with Staking and Distribution

- **Tier module as delegator:** The tier module account holds tier-locked tokens and is the **delegator** in `x/staking` for all tier delegations. The module calls `staking.Delegate`, `staking.Undelegate`, `staking.BeginRedelegate` with itself as delegator.
- **Base rewards:** When the tier module receives base rewards from `x/distribution`, the module updates the cumulative rewards-per-share ratio for the validator (`ValidatorRewardRatio`) and attributes rewards to positions by the ratio delta when they claim. Each position stores a `BaseRewardsPerShare` snapshot; base reward = `DelegatedShares * (currentRatio - BaseRewardsPerShare)`. Distribution withdrawals are deduplicated per validator per block via a transient store. See §5.8 for details.
- **Bonus:** Fixed APY is computed and paid from the tier pool on withdraw; no change to distribution logic.

## 8. Governance voting for tier lockers

Because the **tier module account** is the delegator in `x/staking`, staking-only governance tally attributes all tier-delegated voting power to the module address, not to the tier locker (position owner). Tier lockers would otherwise have no say in governance despite providing security via delegated tier positions. The following changes allow **tier lockers to vote with power proportional to their delegated tier-locked amount**.

### 8.1 Requirement

- Tier lockers vote using their **own address** (standard `MsgVote(proposal_id, option)` from `x/gov`).
- A voter’s **total voting power** = (staking voting power from their direct delegations) **+** (voting power from their **delegated** tier positions).
- Only positions that are **currently delegated** (`Validator` set) count; undelegated or unbonding positions do not count until they are delegated again.

### 8.2 Tier module: voting power API

The tier module must expose at least one of the following so that governance (or the app) can include tier power in the tally:

| Method | Description |
|--------|-------------|
| **GetVotingPowerForAddress(ctx, voterAddr) math.LegacyDec** | Returns the **voting power** (in bond denom units) for the given address: sum of `position.Amount` over all tier positions where `position.Owner == voterAddr` and `position.Validator != ""` (position is delegated). Use the same unit as staking (e.g. bond denom amount) so it can be added to staking voting power. After slashing, `Amount` reflects the current value, so this stays correct. |
| **TotalDelegatedVotingPower(ctx) math.LegacyDec** (optional) | Returns the sum of `Amount` over all delegated positions. Used to include tier-delegated supply in the **total voting power** (quorum denominator) so that quorum and thresholds are consistent. If not implemented, quorum can remain staking-only (tier power only in the numerator). |

Implementations may use `Amount` as the voting power per position (bond denom). If the chain uses validator share-based power for staking, the tier module can instead convert `DelegatedShares` to tokens via the validator’s exchange rate for consistency.

### 8.3 Gov (x/gov) changes

`x/gov` tallies votes using each voter’s **staking** delegations only (`sk.IterateDelegations(ctx, voter, ...)`). The tier module is the delegator for tier positions, so those delegations do not appear under the tier locker’s address. To let tier lockers vote:

- **Custom tally function:** Use the existing extension point **`WithCustomCalculateVoteResultsAndVotingPowerFn`** when constructing the gov keeper. The custom function:
  1. Reuses the default logic to compute each voter’s **staking** voting power and add it to the tally (iterate delegations from the voter, add to results and total).
  2. **For each voter,** calls the tier module’s **`GetVotingPowerForAddress(ctx, voterAddr)`** and adds that **tier voting power** to the same voter’s vote (same weight/options) and to `totalVotingPower`.
  3. Optionally, for **quorum**, the total bonded supply used in the quorum check can include tier-delegated supply: e.g. `totalBonded = sk.TotalBondedTokens(ctx) + tierKeeper.TotalDelegatedVotingPower(ctx)`. If the gov keeper does not have access to the tier keeper, the app can inject a custom tally that uses it and, if needed, a separate mechanism to adjust the quorum denominator (e.g. gov params or a hook that returns “effective total bonded” including tier).

- **Dependency:** Gov does not need to depend on the tier module at compile time; the **app** wires a `CalculateVoteResultsAndVotingPowerFn` that closes over the tier keeper (or an interface) and calls it when tallying. Gov’s `Tally()` continues to use `sk.TotalBondedTokens(ctx)` for quorum unless the app also provides a way to use an effective total that includes tier (e.g. custom tally that returns a different total, or a small change in `Tally` to accept an optional “total bonded” override).

### 8.4 Summary of changes

| Component | Change |
|-----------|--------|
| **Tier module** | Add `GetVotingPowerForAddress(ctx, voterAddr) math.LegacyDec` (sum of `Amount` for delegated positions owned by `voterAddr`). Optionally add `TotalDelegatedVotingPower(ctx)` for quorum. |
| **App / Gov wiring** | Construct gov keeper with `WithCustomCalculateVoteResultsAndVotingPowerFn(fn)` where `fn` adds tier voting power per voter (via tier keeper). Optionally include tier total in quorum denominator. |
| **Voter experience** | Tier lockers vote with **MsgVote** as usual; no new message type. Their voting power automatically includes their delegated tier positions. |

### 8.5 Edge cases

- **Position not delegated:** No tier voting power for that position until the owner delegates via `MsgTierDelegate`.
- **Slashing:** `Amount` is updated on slash (§4.6); tier voting power uses the updated amount.
- **Exit triggered, still delegated:** Position remains delegated until the owner calls `MsgTierUndelegate` and unbonding completes. Until then, it still counts for tier voting power.
- **Unbonding after undelegate:** Once the position’s delegation is cleared (`Validator` empty), it no longer counts. During unbonding, implementation may still have `Validator` set until completion; the ADR leaves that to the implementation (either exclude unbonding positions from `GetVotingPowerForAddress` or clear `Validator` only on unbonding completion).

## 9. Tier-Locked Tokens: Delegate / Undelegate / Redelegate Only via Tier

Tier-locked tokens **cannot** be delegated or redelegated using normal staking messages (undelegation only after the exit commitment has elapsed). They are **internal** to the tier mechanism:

- **MsgLockTier**, **MsgCommitDelegationToTier(tier_id, validator, amount)** (commit partial or full existing delegation to tier without undelegating; amount <= delegation, amount >= MinLockAmount), **MsgAddToTierPosition** (add only when not exiting; reject if exiting or close-only), **MsgClearPosition** (owner cancels exit; settle rewards first; §5.6; rejected if close-only), **MsgTierDelegate(position_id, validator)**, **MsgTierRedelegate(position_id, dst_validator)** are the ways to lock or move tier-locked stake. **MsgTierUndelegate(position_id)** is **only allowed after the user has triggered exit and the exit commitment has elapsed.** Tier lockers cannot voluntarily undelegate while in the tier or during the exit commitment. The tier module account is always the delegator; each position is delegated as a whole (full amount to one validator).
- When a user calls **MsgTierUndelegate** (only valid after exit commitment has elapsed) or **MsgTierRedelegate**, staking runs as usual (unbonding/redelegation). Rewards are claimed before undelegation and redelegation so base + bonus are settled first.
- **Withdraw from tier:** User can trigger exit anytime with **MsgTriggerExitFromTier** (starts exit commitment, per tier's ExitDuration). After exit commitment has elapsed, **MsgTierUndelegate** becomes available and **MsgWithdrawFromTier** claims tokens (no more bonus). If delegated, user must have called **MsgTierUndelegate** (allowed only after exit commitment elapsed) and waited for unbonding before claim.
- **Cancel exit:** **MsgClearPosition** resets exit timestamps so the owner can **add** to the position or **redelegate** again. Rewards are settled first (bonus still capped at `ExitUnlockAt` when applicable) to avoid pool drain (§5.6). Rejected if the tier is in close-only mode.

## 10. Edge Cases and Rules

| Case | Behavior |
|------|----------|
| Lock expired | No bonus; base only. Position can be claimed and closed via **MsgWithdrawFromTier** when §5.7 preconditions are met. |
| No lock | No bonus; base only. |
| Pool empty / insufficient | **User-driven paths** (MsgClaimTierRewards, MsgAddToTierPosition, MsgTierUndelegate, MsgTierRedelegate, MsgClearPosition): if the bonus pool cannot cover the accrued bonus, the message **fails atomically** and state is not mutated. The user can retry after the pool is replenished. **Hook-driven paths** (AfterValidatorBeginUnbonding, BeforeValidatorSlashed): the `LastBonusAccrual` checkpoint is advanced even if the bonus cannot be paid. The unpaid bonus is logged and forfeited so that validator state transitions are not blocked. Base reward withdrawal from distribution is unaffected by the bonus pool balance. |
| Slashing | The position is updated when the delegation is slashed: **Amount** (and **DelegatedShares**) are reduced proportionally to the slash fraction (see §4.6). Base rewards and stake are reduced by distribution/staking. Bonus then accrues on the updated token value. |
| Validator leaves set (jailed, unbonding, removed) | The delegation still exists; the position’s `Validator` and `DelegatedShares` stay valid. No new block rewards while the validator is inactive. User should **MsgTierRedelegate** to an active validator or **MsgTierUndelegate** to unbond. No mandatory position update (see §4.6). Optional: chain may auto-undelegate when a validator is removed; then unbonding-completion update applies. |
| Multiple denoms in base | Base rewards per denom; bonus can be restricted to bond denom or configurable. |
| Trigger exit / claim | User can trigger exit **anytime**. **MsgTriggerExitFromTier** starts exit commitment (wait X years per tier); then **MsgWithdrawFromTier** claims tokens. No bonus after exit commitment elapsed. If delegated, must undelegate and wait unbonding before claim. |
| Multiple positions per owner | Each position is an independent state record; bonus APY and base attribution per position. |
| Add to position when exit triggered | **Reject.** `MsgAddToTierPosition` is not allowed when the position is exiting (`ExitTriggeredAt != 0`); cannot top up while exiting. Use **MsgClearPosition** (owner only) to cancel exit, then add. |
| Clear exit | **MsgClearPosition** clears `ExitTriggeredAt` / `ExitUnlockAt`. Does not withdraw. Settles rewards unconditionally before clearing (§5.6). **Rejected if the tier is in close-only mode.** **MsgWithdrawFromTier** requires exit to be triggered and commitment elapsed; clearing exit revokes that path until **MsgTriggerExitFromTier** is sent again. |
 Lock amount below tier minimum | **Reject.** `MsgLockTier(tier_id, amount, ...)` requires `amount >= tiers[tier_id].MinLockAmount`. If amount is less than the tier’s minimum, the message fails validation. |
| Lock with trigger_exit_immediately but no validator | **Reject.** Current implementation requires `MsgLockTier` to specify a validator, so lock and delegate happen atomically even when `trigger_exit_immediately = true`. |
| Commit delegation: insufficient delegation or below minimum | **Reject.** MsgCommitDelegationToTier requires: signer has a delegation to the specified validator; **amount** ≤ that delegation's token amount; **amount** ≥ tier MinLockAmount; amount > 0. If the user has no delegation to the validator, or amount exceeds their delegation, or amount is below the tier minimum, the message fails. |
| Position not delegated | **No rewards.** Base and bonus rewards accrue **only when the position is delegated** to a validator. Tier positions created by `MsgLockTier` and `MsgCommitDelegationToTier` start delegated; a position only becomes undelegated after a valid `MsgTierUndelegate` and its unbonding lifecycle. There is no voluntary "undelegated and idle" state before exit commitment elapses. |

## 11. Security and Invariants

- **Authority:** Only designated authority (e.g. gov) can update tier params and fund the pool.  
- **Tier minimum lock:** For each tier, `MinLockAmount` is enforced on **MsgLockTier** and **MsgCommitDelegationToTier**: the locked or committed amount must be ≥ the tier's MinLockAmount. Params must set `MinLockAmount` ≥ 0; implementations may require it to be positive for a tier to accept new locks.  
- **Commit delegation verification:** **MsgCommitDelegationToTier(tier_id, validator, amount)** requires amount > 0, amount ≥ tier MinLockAmount, and amount ≤ signer's delegation to that validator. Partial commit is allowed. One message creates one tier position (one validator; the position itself cannot be split across validators).  
- **Rewards only when delegated:** Tier lockers receive **no** base or bonus rewards unless the position is delegated to a validator. This enforces that lockers contribute to network security to earn rewards.  
- **No undelegation until exit commitment elapsed:** **MsgTierUndelegate** is only allowed when the exit commitment has elapsed (`block_time >= ExitUnlockAt`). Tier lockers cannot undelegate while in the tier or during the exit commitment period, so they keep providing security until the commitment is over.
- **No double bonus:** Bonus is fixed APY accrued over time; tracked per position (`LastBonusAccrual`); paid on `MsgClaimTierRewards` (and before TierUndelegate/TierRedelegate/AddToPosition/ClearPosition). **MsgClearPosition** must settle rewards **before** clearing exit timestamps so accrual cannot be extended past `ExitUnlockAt` without the exit cap (see §5.6).
- **Pool balance:** Never send more than pool balance. User-driven paths fail atomically if the pool is insufficient. Hook-driven paths advance the checkpoint and forfeit unpaid bonus.
- **CloseOnly tiers:** When a tier is marked close-only, new positions, additions, delegations, redelegations, and clear-position are blocked. Only exit, undelegate, withdraw, and claim remain available.
- **Tier-only delegation:** Only tier module can delegate/undelegate/redelegate tier-locked tokens; staking messages from users do not affect tier positions.  
- **Withdraw address:** Base rewards attributed to positions and sent to owners; bonus paid to position owner.

## 12. Optional: Distribution Hook for Base Attribution

Because the **tier module** is the delegator in staking, base rewards are paid to the tier module. To attribute base rewards to individual positions (each position is one delegation, so attribution is per position), the chain can:

- Use a single withdraw address (the tier module account) and attribute base rewards internally by position when users call `MsgClaimTierRewards`, or  
- Extend `x/distribution` with a hook (e.g. `AfterDelegationRewardsWithdrawn`) so the tier module can attribute and forward base to position owners when distribution pays out.  

Bonus is **fixed APY** from the tier pool, not a multiplier on base; the hook would only affect how **base** rewards are attributed, not the bonus formula.

## 14. Summary

- **Tier = lock duration + fixed bonus APY + minimum lock amount + close-only flag.** Each tier defines `ExitDuration`, `BonusApy`, `MinLockAmount`, and `CloseOnly`. Tiers are managed via governance messages (`MsgAddTier`, `MsgUpdateTier`, `MsgDeleteTier`), stored separately from params. Module params contain only `TargetBaseRewardsRate`. **MsgLockTier** (with required validator and optional `trigger_exit_immediately`) and **MsgCommitDelegationToTier(tier_id, validator, amount)** (commit partial or full existing delegation without undelegating) create tier positions. **Rewards (base and bonus) are paid only when the position is delegated to a bonded validator**. **Undelegation is only allowed after the user has triggered exit and the exit commitment has elapsed**. **MsgClearPosition** lets the owner cancel exit (settle rewards first -- §5.6) so they can add to the position or redelegate again. Tier-locked tokens are **internal**: **MsgTierDelegate**, **MsgTierRedelegate**, **MsgTierUndelegate** (after exit commitment elapsed) move stake; the tier module account is the delegator. **MsgCommitDelegationToTier** requires a chain extension to transfer delegation (same validator, no unbonding).
- **Tier positions are state records:** each lock is a `Position` (id, owner, tier_id, amount, created_at_height, created_at_time, exit_triggered_at, exit_unlock_at, validator, delegated_shares, base_rewards_per_share, last_bonus_accrual). A position cannot be broken down; the full amount is delegated to one validator. The owner can **add** to an existing position via `MsgAddToTierPosition` **only when not exiting** (reject if exiting). **MsgClearPosition** cancels exit (settle rewards unconditionally first -- §5.6). Store: `Positions`, `PositionsByOwner`, `PositionsByTier`, `PositionsByValidator`, `PositionCountByTier`, `ValidatorRewardRatio`, unbonding/redelegation mappings.
- **Withdraw from tier:** User can trigger exit **at any time**. **MsgTriggerExitFromTier** starts the exit commitment (wait per tier’s `ExitDuration`). Once it has elapsed, **no more bonus** and **MsgWithdrawFromTier** claims tokens (after unbonding if delegated).
- **Bonus = fixed APY** on delegated token value: constant BonusApy; accrued and paid from a **tier rewards pool** when user calls `MsgClaimTierRewards` (and before TierUndelegate/TierRedelegate/AddToPosition/ClearPosition). Bonus stops at ExitUnlockAt. No bonus when validator is not bonded.
- **Base rewards:** Attributed via cumulative rewards-per-share (`ValidatorRewardRatio` per validator, `BaseRewardsPerShare` snapshot per position). Distribution withdrawals deduplicated per validator per block. Optional `TargetBaseRewardsRate` top-up in BeginBlocker.
- **Integration:** Tier module holds tokens and delegates via staking; base rewards received by module and attributed to positions via ratio; bonus from pool.
- **Governance voting:** Tier lockers vote with their address; voting power includes their delegated tier positions via a custom gov tally and `VotingPowerByOwner` / `TotalDelegatedVotingPower` queries (§8).
- **External pool:** Filled by governance or another module; tier module pays fixed-APY bonus from it.
- **Safety:** User-driven paths fail atomically if pool is insufficient; hook paths forfeit unpaid bonus. Only tier module can delegate/undelegate/redelegate tier-locked tokens. Close-only tiers prevent new positions while allowing existing ones to close out.
